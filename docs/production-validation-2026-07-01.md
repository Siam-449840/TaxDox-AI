# TaxDox AI Production Validation Report

Date: 2026-07-01

## Executive Summary

TaxDox AI is not certified production-ready in this environment.

The codebase now passes static validation, dependency audit, Prisma schema validation, and production build. Several verified security blockers were remediated during this pass, especially cross-tenant IDOR risks in API routes, unsanitized Word preview HTML, and production query logging of sensitive data.

Runtime production certification is blocked by missing production infrastructure/configuration:

- `ENCRYPTION_KEY` was absent from `.env`; production boot correctly failed closed until a temporary process-only key was injected.
- Postgres at `localhost:5432` was unreachable; Docker daemon was unavailable, so the local DB container could not be started.
- Redis/Upstash was not configured; production login rate limiting correctly failed closed, preventing authenticated smoke tests.
- R2, Resend, Inngest, Sentry, and alerting credentials were not configured, so storage, email, queue, monitoring, and alert verification could not be completed.

Go/No-Go: **NO-GO** until the blocked infrastructure checks are completed in staging/production with real credentials.

## Evidence Collected

Static/build:

- `npm run lint`: passed.
- `npx tsc --noEmit`: passed.
- `npx prisma validate`: passed.
- `npm audit --audit-level=moderate`: passed, 0 vulnerabilities.
- `npm run build`: passed on Next.js 16.2.9.
- Build artifact sizes: `.next` 429 MB, `.next/standalone` 127 MB, `.next/static` 2.5 MB.

Runtime:

- Production boot without `ENCRYPTION_KEY`: failed closed with `[env] Refusing to boot`.
- Production boot with temporary in-process `ENCRYPTION_KEY`: server started.
- `GET /api/health/live`: 200.
- `GET /api/health/ready`: 503; database down and Redis missing.
- Unauthenticated protected APIs returned 401: `/api/dashboard`, `/api/reports`, `/api/documents`, `/api/emails`, `/api/notifications`, `/api/audit-logs`.
- CSRF tests returned 403 for mutating `/api/clients` with no origin and with evil origin.
- Fake/tampered session cookie returned 401.
- Security headers present on `/auth/signin`: HSTS, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, permissions policy, CSP report-only.
- 100-VU k6 liveness load: 37,347 requests in 20s, 0% failures, P95 92.88 ms, P99 not exported by custom summary but max was 248.82 ms.

Blocked:

- Authenticated smoke failed because production rate limiting failed closed without Redis.
- DB-backed API latency, upload latency, extraction time, queue latency, backup/restore, and realistic load tests were blocked by unavailable Postgres/Redis/R2/Inngest.
- Monitoring alert delivery could not be verified because Sentry/alerting was not configured.

## Remediations Applied

Security/access control:

- Added tenant authorization to dashboard, reports, audit logs, document list/detail/update/delete, extraction read/update, engagement detail/update, PBC list/item routes, emails, notifications, and AI classify/extract endpoints.
- Added ownership checks for upload `engagementId` and `pbcItemId`, not just `clientId`.
- Added signed internal header support for background extraction calls so queue/inline jobs do not require a browser session while public AI calls remain session and tenant checked.
- Sanitized Word preview HTML generated by Mammoth before returning it to the frontend.
- Disabled Prisma query logging in production to reduce PII leakage.

Supply chain:

- Updated vulnerable direct dependencies.
- Added controlled overrides for transitive `postcss` under Next and `uuid` under NextAuth.
- Re-ran audit to 0 vulnerabilities.

Build/reliability:

- Fixed k6 lint warning.
- Added Turbopack trace-ignore annotations for dynamic local object-store paths.

## Security Audit Results

Passed locally:

- Authentication bypass: protected APIs returned 401 without session.
- CSRF: mutating API request with missing/mismatched origin returned 403.
- JWT/session tampering: fake cookie returned 401.
- Dependency vulnerabilities: 0 moderate-or-higher vulnerabilities after remediation.
- Security headers: baseline headers present.
- XSS: verified unsafe Word-preview path and remediated with server-side sanitization.
- Secret scan: no live key pattern found in source; findings were placeholders/docs/examples and code variable names.

Not fully verified:

- Cross-tenant isolation after login: code was remediated, but authenticated multi-tenant runtime test was blocked by Redis/DB.
- SQL injection: Prisma usage is generally structured; raw SQL is limited to health probes. DB-backed injection probes were blocked.
- File upload abuse/MIME spoofing: code review found MIME allowlist and size limit; runtime upload probes were blocked by auth/DB.
- SSRF: no obvious arbitrary URL fetch surface found in audited API routes.
- Rate-limit behavior: production fail-closed verified indirectly via login logs; Redis-backed allow/deny windows require Upstash.
- Sentry/alerting: not configured.
